# MacLocalFileManager v2.0 Core Spotlight Framework

## 1. Positioning

v2.0 moves semantic search from an app-owned experimental vector layer to a system-backed Core Spotlight layer.

The product remains a no-online-AI local search tool. Core Spotlight is used as an on-device semantic search backend, not as a replacement for the app's controlled SQLite ranking.

Primary goals:

- Keep Chinese filename precision as the highest-priority search path.
- Use Core Spotlight semantic search for natural-language supplemental results.
- Reuse system indexing models for text and media where available.
- Avoid shipping large local model files in the app package.
- Keep all file operations local, previewable, and non-destructive.

Non-goals:

- Do not build a chat assistant.
- Do not use online LLM APIs.
- Do not replace SQLite exact search with Spotlight ranking.
- Do not depend on Finder or global Spotlight result ordering.
- Do not expose private indexed content to other apps.

## 2. Architecture Decision

v2.0 should use a hybrid search architecture:

1. SQLite remains the source of truth for file metadata, category, existence state, and deterministic ranking.
2. Core Spotlight receives donated searchable items for semantic retrieval.
3. A Swift helper owns Core Spotlight calls.
4. PySide6 talks to the helper through JSON over subprocess.
5. Search results are merged in Python.

Reasoning:

- Core Spotlight APIs are native Apple APIs with async Swift-first ergonomics.
- PyObjC can call parts of Core Spotlight, but `CSUserQuery`, async responses, preparation, batching, and future delegate integration are less maintainable through Python.
- A helper process isolates Core Spotlight runtime behavior from PyInstaller packaging and keeps the Python UI stable.

## 4. Search Flow

### 4.1 Query Path

```text
User input
  -> FileSearcher.search()
    -> SQLite exact/fuzzy/content/OCR search
    -> optional Core Spotlight semantic query
    -> merge by file path / file id
    -> deterministic final ranking
    -> UI
```

### 4.2 Ranking Policy

Final ordering remains controlled by the app:

1. Exact filename match.
2. Filename starts with query.
3. Filename contains query.
4. Filename fuzzy match.
5. Path contains query.
6. Content/OCR keyword match.
7. Core Spotlight semantic result.
8. Local fallback semantic/vector result.

Core Spotlight may return ranked results, but those ranks are only used inside the semantic band. They must not outrank exact filename hits.

## 5. Indexing Flow

### 5.1 SQLite Index

SQLite continues to store:

- `id`
- `filename`
- `normalized_filename`
- `path`
- `parent_dir`
- `extension`
- `size`
- `created_at`
- `modified_at`
- `indexed_at`
- `exists`
- category metadata
- content/OCR state

### 5.2 Core Spotlight Donation

The app donates searchable items for files that are safe and useful:

- Text documents: title, display name, content text, path, content type.
- PDFs: title, extracted text if available, path, content type.
- Images: title, path as `contentURL`, content type.
- Engineering files: title, path, category metadata, extension-specific UTType where possible.

The donation unique identifier should be deterministic:

```text
mlfm::<file_id>::<normalized_path_hash>
```

Path remains the merge key because SQLite id may differ across machines.

### 5.3 Incremental Donation

Only donate when one of these changes:

- file path
- file size
- modified timestamp
- extracted text version
- category
- content type mapping
- helper schema version

Use a dedicated state table:

```sql
CREATE TABLE IF NOT EXISTS spotlight_items (
    file_id INTEGER PRIMARY KEY,
    unique_identifier TEXT NOT NULL UNIQUE,
    path TEXT NOT NULL,
    content_type TEXT NOT NULL,
    source_size INTEGER NOT NULL,
    source_modified_at REAL NOT NULL,
    donated_at REAL NOT NULL,
    donation_version INTEGER NOT NULL,
    status TEXT NOT NULL,
    error TEXT NOT NULL DEFAULT '',
    FOREIGN KEY(file_id) REFERENCES files(id)
);
```

## 6. Swift Helper Contract

The helper is a local CLI executable bundled inside the app.

### 6.1 Commands

```bash
spotlight-helper capabilities
spotlight-helper prepare
spotlight-helper donate --json-lines
spotlight-helper delete --json-lines
spotlight-helper search --query "海边图纸" --limit 50 --json
spotlight-helper reset-domain --domain local.maclocalfilemanager
```

## 7. Python Integration

Add `spotlight_bridge.py`:

```python
class SpotlightBridge:
    def capabilities(self) -> SpotlightCapabilities: ...
    def prepare(self) -> SpotlightCommandResult: ...
    def donate(self, items: Iterable[SpotlightDonationItem]) -> SpotlightCommandResult: ...
    def delete(self, items: Iterable[SpotlightDeleteItem]) -> SpotlightCommandResult: ...
    def search(self, query: str, limit: int, category: str) -> list[SpotlightResult]: ...
```

Bridge requirements:

- All helper calls have timeouts.
- Failure never breaks SQLite search.
- stderr is captured into app logs.
- JSON parse errors are surfaced as disabled Spotlight state.
- Search calls are cancellable by discarding stale search ids.

## 8.1 Appearance Framework: Frosted and Liquid Glass

v2.0 should add Liquid Glass as an appearance option, not as the only UI mode.

Apple's current design guidance treats Liquid Glass as a functional layer for controls and navigation that floats above content. It is not intended as a blanket background behind all content. The search app should therefore use Liquid Glass only for the top-level interactive search shell, compact filter controls, and optional floating action groups.

### Modes

| Mode | Backend | Target |
| --- | --- | --- |
| `system` | Qt palette + current macOS appearance | Default safe mode |
| `frosted` | PySide translucent stylesheet + optional `NSVisualEffectView` host | Current beta-compatible visual mode |
| `liquidGlass` | AppKit `NSGlassEffectView` / `NSGlassEffectContainerView` through Swift/AppKit helper or native wrapper | v2.0 native macOS mode |

### Design Rules

- Use glass for the search field shell and primary floating controls.
- Do not put long text tables directly on Liquid Glass.
- Keep result tables on a calmer frosted/content layer for readability.
- Group nearby glass controls into one glass container to avoid inconsistent sampling and improve performance.
- Do not make non-interactive labels look like buttons.
- Preserve the magnifying-glass/search-field mental model.
- Respect Reduce Transparency and Increase Contrast accessibility settings.
- Fall back to `frosted` if native glass APIs are unavailable.

### Proposed UI Layering

```text
Window background
  -> content layer
    -> result table / image grid / detail preview
  -> frosted layer
    -> result container backing
  -> Liquid Glass functional layer
    -> search field
    -> filter chips
    -> compact command buttons
```

### PySide6 Constraint

PySide6 stylesheets can approximate frosted glass with translucent backgrounds, rounded corners, and low-contrast borders, but they cannot produce true system Liquid Glass optical behavior.

For v2.0, there are two implementation paths:

1. Short-term: keep PySide6 and expose `frosted` as the reliable default.
2. Native path: add an AppKit wrapper for the search shell using `NSGlassEffectView`, or move the shell/window to Swift/AppKit while keeping Python as the indexing/search engine.

### Settings Schema

Add:

```text
appearance_mode = system | frosted | liquidGlass
appearance_reduce_transparency = auto | on | off
appearance_glass_tint = none | accent | subtleBlue | graphite
```

Default:

```text
appearance_mode = system
appearance_reduce_transparency = auto
appearance_glass_tint = none
```

### Acceptance Criteria

- Switching appearance mode does not restart indexing.
- `liquidGlass` mode falls back to `frosted` when native APIs are unavailable.
- Result table remains readable in light and dark appearance.
- Search field remains the most visually prominent control.
- Accessibility settings can override transparency-heavy modes.

## 9. macOS Compatibility

v2.0 should define capability bands instead of assuming one OS behavior:

| Capability | Behavior |
| --- | --- |
| Core Spotlight unavailable | Disable system semantic backend, keep SQLite search |
| Core Spotlight lexical only | Use helper for structured search only if useful |
| Semantic available | Enable semantic supplemental results |
| Ranked results available | Use rank only inside semantic band |
| Prepare failed | Defer semantic search and show recoverable status |

The app must start and search files even if every Core Spotlight feature fails.

## 10. Privacy and Safety

Rules:

- Only donate files already indexed by MacLocalFileManager.
- Do not donate excluded paths.
- Do not donate `/System`, `/Library`, `~/Library`, `/Applications`, or installer volumes.
- Do not donate file content for files larger than configured limits.
- Do not delete user files.
- Deleting from Spotlight means deleting app donations only.
- Keep SQLite `exists = 0` behavior unchanged.

## 11. Implementation Phases

### Phase 0: Feasibility PoC

Goal: prove Core Spotlight semantic search works on this Mac with donated test files.

Deliverables:

- Minimal Swift CLI.
- Donate 20 fixture items.
- Query with natural-language text.
- Verify returned identifiers.

Acceptance:

- Helper can donate text/PDF/image metadata.
- Helper can query and return JSON.
- App can run if helper is missing.

Estimated Codex token cost: 30k-50k.

### Phase 1: Helper Package

Goal: production-shape helper with stable command contract.

Deliverables:

- `spotlight_helper/Package.swift`
- JSON payload structs.
- Donation service.
- Search service.
- Capability detection.
- Unit tests where possible.

Acceptance:

- Helper builds from script.
- Helper returns deterministic JSON errors.
- Helper is bundled by PyInstaller.

Estimated Codex token cost: 35k-55k.

### Phase 2: Python Bridge

Goal: integrate helper without changing UI search behavior.

Deliverables:

- `spotlight_bridge.py`
- Settings defaults.
- Timeout/error logging.
- Tests with fake helper executable.

Acceptance:

- Fake helper tests pass.
- SQLite search still works if helper returns errors.
- No UI blocking.

Estimated Codex token cost: 25k-45k.

### Phase 3: Donation Pipeline

Goal: donate indexed files incrementally.

Deliverables:

- `spotlight_items` table.
- Incremental donation after file scan/content scan.
- Failed donation retry.
- Clear/rebuild command.

Acceptance:

- Repeated scan does not re-donate unchanged files.
- Missing files remove app donations.
- Progress is visible.

Estimated Codex token cost: 35k-60k.

### Phase 4: Hybrid Search

Goal: merge Spotlight semantic results with app-ranked SQLite results.

Deliverables:

- Spotlight search branch in `FileSearcher`.
- Dedupe by path.
- Rank band policy.
- Result source badges.

Acceptance:

- Exact filename match always beats Spotlight semantic result.
- Semantic results appear for natural-language queries.
- Category filters still work.

Estimated Codex token cost: 30k-55k.

### Phase 5: Packaging and Runtime Validation

Goal: shipable app package.

Deliverables:

- Build helper during packaging.
- Bundle helper into `.app`.
- Verify `.dmg`.
- Runtime tests on a clean data directory.

Acceptance:

- `.app` launches without terminal.
- Missing helper fails gracefully.
- Helper works inside packaged app.

Estimated Codex token cost: 30k-60k.

### Phase 6: Native Appearance Layer

Goal: add Liquid Glass as a system-native appearance option.

Deliverables:

- Appearance setting and persistence.
- PySide frosted fallback.
- AppKit prototype for `NSGlassEffectView` search shell.
- Native capability detection.
- Screenshot regression tests for collapsed and expanded search states.

Acceptance:

- Users can switch between System, Frosted, and Liquid Glass modes.
- The app remains usable on macOS versions without Liquid Glass APIs.
- Glass is limited to the search shell and floating controls.
- Result readability is not reduced.

Estimated Codex token cost: 35k-70k.

## 13. Risks

### Risk: Semantic search may not produce useful results for arbitrary file paths

Mitigation:

- Donate extracted text and high-quality metadata.
- Keep SQLite exact search first.
- Add settings to disable system semantic.

### Risk: Model availability is system-controlled

Mitigation:

- Helper exposes `prepare`.
- UI shows readiness.
- Search degrades to SQLite.

### Risk: Core Spotlight indexing behavior differs by macOS version

Mitigation:

- Capability detection.
- Small PoC before broad integration.
- Keep local semantic fallback until v2.0 is stable.

### Risk: Python packaging with Swift helper adds complexity

Mitigation:

- Build helper separately.
- Bundle as a data/binary file.
- Keep subprocess JSON contract stable.
